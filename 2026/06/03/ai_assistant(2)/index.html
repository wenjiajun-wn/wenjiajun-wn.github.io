  • AI 总结：一张截图 → 一段 80 字内的简洁总结
  • 截止日识别：自动从总结中提取日期（6月20日 / 明天 / 下周五等），放到对应日历格子
  • 日历网页：月/周/日视图，左侧边栏当天事项，全屏自适应

  • 浏览器唤醒：首次推送自动打开日历网页
  • 优先级切换：点击圆点循环切换 普通/重要/紧急，颜色+标签同步
  • 批量删除：侧边栏复选框 + 删除所选
  • 去重防重复：哈希 + 尺寸去重 + 5秒冷却 + 单实例文件锁

完整流程：Win+Shift+S 截图选区域 → AI 一句总结 → 提取截止日 → 自动推送到日历 → 浏览器弹出显示。

关键技术概念

哈希

哈希是一种将任意长度的数据（比如文件、文本、数字）转换成固定长度短字符串的算法。这个短字符串通常被称为哈希值，相当于给数据一个独一无二的”指纹”——相同的数据永远返回相同的哈希值，而且计算速度很快。

哈希的主要用途：

  • 数据去重：比较哈希值判断内容是否重复
  • 密码储存：只存哈希值，不存明文密码
  • 文件完整性校验：对比下载前后哈希值是否一致
  • 哈希表/字典：编程语言中的 Map、Object、Dict 都用哈希来快速定位键值对

单实例文件锁

单实例文件锁是一种并发控制/互斥机制，确保同一时间整个系统只有一个进程（或线程）在执行核心的去重+处理逻辑。它使用操作系统提供的文件锁（flock、LockFile 等）来锁住一个约定的锁文件。

进程启动时尝试获取锁：

  • 成功获取 → 成为单实例，继续执行
  • 获取失败 → 说明已经有另一个实例在运行，当前进程直接退出或等待

在分布式或多进程环境下，文件锁能防止多个进程同时修改状态、同时处理同一个内容，导致重复处理或数据错乱。它是一种轻量级的单实例保障。

防重复流程

一个典型的完整防重复流程如下：

  1. 获取单实例文件锁
    确保同一时间只有一个处理实例在运行。如果拿不到锁，直接退出。

  2. 检查尺寸 + 哈希
    计算文件的尺寸和哈希值，在去重记录表（比如 Redis、数据库、本地缓存）中查找是否存在该组合。

    • 若存在 → 判定为重复，结束处理
    • 若不存在 → 记录该尺寸+哈希，然后继续处理

  3. 5秒冷却检查
    对于刚刚处理过的内容，在去重记录表里额外记录一个”处理时间戳”。如果当前时间与上次处理时间差 < 5 秒，则直接拒绝（即使尺寸+哈希相同）。
    实际中常把冷却时间整合到去重记录的过期时间中（例如键值对设置 TTL = 5 秒）。

  4. 执行业务处理（如下载、入库、通知等）

  5. 更新冷却标记
    设置该内容的最后处理时间为当前时间，并保留 5 秒。

新增多用户支持

数据层上，JSON 结构从 [{event}] 升级为 {"events":{"userA":[],"userB":[]}}，旧数据自动迁移。API 加上 ?user= 参数隔离不同用户。

前端 JS 添加 USER_ID 常量，fetch() URL 追加 ?user=，页面标题显示用户名。

客户端 hotkey_screenshot.py 读取 user_config.json，推送和浏览器 URL 带用户参数。

打包与部署

  • **build_exe.bat**：PyInstaller 一键打包脚本
  • **user_config.json**：用户配置模板（用户名 + 服务器地址）
  • .env 模板：不含 API Key，用户自己申请

部署使用阿里云轻量服务器，通过 Termius 连接，SFTP 上传文件。

数据库

使用 SQLite 轻量数据库（Python 内置），替换了原来的 JSON 存储方案。新增两个核心函数：

  • **_get_db()**：创建数据库连接，开启 WAL 模式（允许并发读写）
  • **_init_db()**：建表 + 检测旧的 calendar_data.json 自动迁移

自动迁移机制：_init_db() 启动时会检查是否存在旧文件 calendar_data.json。如果存在，自动把数据导入 SQLite，然后把 JSON 文件重命名为 calendar_data.json.migrated，不会重复迁移。

新增登录界面（/login）

新增 users 表 + 注册/登录 API：

方法 路径 说明
POST /api/auth/register 注册：输入用户名+密码，创建账号
POST /api/auth/login 登录：验证密码，写入 Session
POST /api/auth/logout 登出：清除 Session
GET /api/auth/me 查询当前登录状态

Session 数据存在服务端（Flask 默认存在加密的 Cookie 里，也可以存 Redis/数据库），浏览器只持有一个 Session ID（一串随机字符），不存实际数据，因此安全。

权限控制：

  • 没登录 → 自动跳转到 /login
  • 已登录 → 显示自己的日历

exe 客户端怎么认证？

浏览器用 Cookie/Session，exe 用 API Token：

  1. 注册成功后会给一个 32 位随机 Token，写在 user_config.json 里
  2. exe 每次请求在 HTTP Header 里带上 X-API-Token: xxx
  3. 服务端就能识别是谁在推送

用户使用流程

首次使用

  1. 打开网页 → 自动跳转登录页 → 点击”注册”
  2. 输入用户名 + 密码 → 注册成功
  3. 复制 API Token → 填到 exe 的 user_config.json 中
  4. 之后打开网页就自动登录了

拿到 exe 后的操作：

  1. 双击 AI-TODO-Assistant.exe
  2. 首次运行弹出提示，输入用户名 + 密码
  3. 自动尝试登录 → 没账号就自动注册 → Token 自动保存
  4. 窗口显示 “Setup complete! Starting…”
  5. 按 Win+Shift+S 截图即可使用

日常使用

  • 打开网页 → 已登录，直接看到日历
  • 截图 → exe 用 Token 推送 → 日历自动刷新
