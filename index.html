什么是 FastAPI？

FastAPI 是一个现代、高性能的 Python Web 框架，专门用于构建 API。它由 Sebastián Ramírez 开发，基于以下两个强大的库：

  • Starlette：提供异步 Web 框架的核心能力
  • Pydantic：负责数据验证与序列化

FastAPI 的名字本身就说明了它的两大特点：Fast（快速）—— 既指运行速度快，也指开发速度快。

为什么选择 FastAPI？

第二节：FastAPI 进阶主题

一、PUT 与 DELETE 操作

在第一节里，我们学会了用 GET 读取数据、用 POST 创建数据。现在有另外两种 HTTP 操作：

  • PUT：更新已有对象
  • DELETE：删除已有对象

和 POST 一样，它们都可以从查询字符串（query string）和请求体（request body）中接收参数。关键区别在于——这两种操作需要引用一个已存在的对象。

引用已有对象

三、使用 async 处理并发请求

现代 API 往往需要同时处理大量请求。FastAPI 支持 Python 的 async/await 语法，让你的应用能以更低的消耗应对更高的并发。

什么时候用 async？

  • 调用外部 API 或数据库时（I/O 密集型操作）
  • 操作本身有等待时间，而 CPU 其实处于空闲状态时

如果函数只做纯 CPU 计算（不涉及 I/O 等待），则不需要 async。

异步 DELETE 功能

加上 async def 和 await，FastAPI 就能在等待数据库响应时继续处理其他请求，显著提升吞吐量。

用 Hugging Face smolagents 打造智能 AI Agent

前言

AI Agent（AI 代理）正在改变我们与软件互动的方式。过去，我们给 LLM 一个问题，它回答；现在，Agent 能够搜索数据、推理、执行代码，甚至主动完成复杂任务。本篇博客整理 smolagents 框架的四大核心主题并学习多代理系统的设计

一、Introduction to Hugging Face smolagents

具体网址：Hugging Face Hub 文档

什么是 smolagents？

smolagents 是 Hugging Face 开发的轻量级 Python 框架，专为构建能够推理与行动的 AI Agent 而设计。它的核心理念是让 LLM 不只是”回答问题”，而是能够生成并执行 Python 代码来完成任务。

两种核心 Agent 类型

类型 说明 行动方式
CodeAgent smolagents 的主要 Agent 类型 生成并执行 Python 代码
ToolCallingAgent 第二种 Agent 类型 输出 JSON/文本格式，由系统解析后执行

CodeAgent 的优势在于它直接输出可执行的代码，逻辑更灵活、更强大；ToolCallingAgent 则更适合需要严格结构化输出的场景。

Tools（工具）：Agent 的行动工具

Tool是 Agent 能够调用的函数，是 Agent 的基本组成。smolagents 提供三种方式定义工具：

  1. 内置工具箱：如 DuckDuckGoSearchTool、VisitWebpageTool 等DuckDuckGoSearchTool：调用 DuckDuckGo 搜索引擎，帮 Agent 上网搜东西。相当于 Agent 的”百度一下”，不用 API Key，免费用
    VisitWebpageTool：打开一个网页，把里面的文字内容抓回来。搜到结果后点进去看具体内容，相当于 Agent 的”打开网页看一眼”
  2. @tool 装饰器：把一般函数变成 Agent 可调用的工具
  3. 继承 Tool 类：更完整的自定义工具实现

工具也可以分享到 Hugging Face Hub，或加载社区贡献的工具，形成开放的工具生态系统。

为什么选择 smolagents？

  • 轻量：核心逻辑精简，容易理解与修改
  • 与模型无关：支持 HF Inference、OpenAI、Ollama、vLLM 等多种后端，一行切换
  • 安全执行：代码可以在 Docker、E2B、WebAssembly sandbox 沙箱中隔离执行
  • 生态完整：深度整合 Hugging Face Hub，工具与模型唾手可得

二、Agenti RAG and Multi-Step Agents

传统的 RAG 限制

传统 RAG 的流程是：一次查询 → 一次检索 → 生成答案。这在面对复杂问题时容易出现”检索不够精准”或”信息不足”的困境，因为它无法根据中间结果动态调整搜索策略。

Agent RAG：让 Agent 主动检索

Agentic RAG 赋予 Agent 自主决策的能力：

  • 自主重构查询：根据问题语义，自动生成更精确的搜索词
  • 迭代式检索：分多步骤从不同来源获取信息
  • 批判性评估：自动判断检索结果是否足够，决定是否再次搜索
  • 多来源整合：同时结合网络搜索与本地知识库

规划间隔（Planning Intervals）

在多步骤任务中，可以设定让 Agent 定期”停下来想一想”——评估目前进度、调整接下来的行动策略，避免一直盲目执行而偏离目标。

回溯机制(Callbacks)：实时监控 Agent 行为

通过 Callback 机制，可以在 Agent 执行的每个步骤插入自定义逻辑，用于：

  • 实时追踪执行状态
  • 记录 Token 使用量与延迟
  • 在特定条件下中断或修改行为

Stateful Tools（有状态工具）

有别于一般无状态函数，Stateful Tool 能在多步骤任务中维持内部状态，例如记住上一步的检索结果，让后续步骤可以接续使用，大幅提升复杂任务的效率。

三、Multi-Agent Systems

为什么需要多代理系统？

单一 Agent 处理复杂任务时能力有限，以及 Context 窗口有限。多代理系统通过分工合作解决这个问题：把复杂任务拆分给专业 Agent，各司其职，再由 Manager Agent 协调整合。

架构设计：Manager(管理) + Specialist(专家)

Manager Agent 负责拆解任务、指派子任务给各专业 Agent，并整合最终结果。

典型的多代理 RAG 系统

一个完整的多代理 RAG 系统可能包含：

  • Web Agent：负责搜索网络最新信息
  • Retriever Agent：查询本地知识库或向量数据库
  • Image Generation Agent：按需求生成图像
  • Manager Agent：整合所有子 Agent 的输出

这种架构适合需要跨领域知识整合的复杂任务。

四、Memory and Validation

Memory：让 Agent 记住对话的上下文

没有记忆的 Agent 每次对话都是”从零开始”，无法处理追问或多轮任务。smolagents 的记忆系统让 Agent 能够：

  • 保留对话历史，自然处理追问
  • 记住已完成的步骤，避免重复工作
  • 根据过往互动调整策略

Memory 以步骤形式序列化存储，包含时间戳与 Token 使用量等元数据，方便后续分析。

执行追踪（Execution Traces）

除了记忆，smolagents 还提供详细的执行追踪：

  • 推理步骤记录：完整保留 Agent 每一步的思考过程
  • Debug 辅助：当 Agent 行为异常时，可以逐步回放推理过程找出问题
  • 性能分析：了解哪些步骤耗时、哪些工具调用最频繁

Validation（输出验证）

Agent 的输出不一定永远正确，验证机制是确保可靠性的关键环节：

常见验证方法：

  1. 结果相关性验证：确认检索到的信息确实回答了问题
  2. 格式验证：确保输出符合预期的数据结构
  3. 事实一致性检查：交叉比对多个来源的信息
  4. Fallback 机制：当主要方法失败时，启动备用策略

错误恢复：

smolagents 内置错误日志与重试机制，让 Agent 在遇到工具调用失败或结果不佳时，能自动尝试替代方案，而不是直接中断任务。

总结：smolagents 的核心设计哲学

主题 核心概念
基础框架 CodeAgent 以 Python 代码为行动语言，Tool 是行为积木
Agentic RAG 迭代式、多步骤的智能检索，取代一次性的静态查询
多代理系统 Manager 协调 Specialist，分工合作处理复杂工作流
记忆与验证 跨轮次记忆 + 执行追踪 + 验证策略，确保可靠输出

smolagents 的设计是框架本身保持轻量，让开发者能够完全理解其运作原理，同时通过工具生态与多代理协作，支撑起现实中的复杂应用场景。

无论是自动化数据的工作流还是智能文档问答系统，或者跨领域的研究助理，smolagents 都可以提供一个清晰、灵活、可扩展的起点。

特点：

  • 存在服务器端（Flask 默认加密存在 Cookie 里，也可以存 Redis/数据库）

Cookie + Session 的配合流程：

  4. 之后每次请求，浏览器自动在 Cookie 里带上 Session ID
  5. 服务器根据 Session ID 查到 Session 数据 → 知道是谁在操作

关键技术概念

哈希

哈希是一种将任意长度的数据（比如文件、文本、数字）转换成固定长度短字符串的算法。这个短字符串通常被称为哈希值，相当于给数据一个独一无二的”指纹”——相同的数据永远返回相同的哈希值，而且计算速度很快。

哈希的主要用途：

  • 数据去重：比较哈希值判断内容是否重复
  • 密码储存：只存哈希值，不存明文密码
  • 文件完整性校验：对比下载前后哈希值是否一致
  • 哈希表/字典：编程语言中的 Map、Object、Dict 都用哈希来快速定位键值对

单实例文件锁

单实例文件锁是一种并发控制/互斥机制，确保同一时间整个系统只有一个进程（或线程）在执行核心的去重+处理逻辑。它使用操作系统提供的文件锁（flock、LockFile 等）来锁住一个约定的锁文件。

进程启动时尝试获取锁：

  • 成功获取 → 成为单实例，继续执行
  • 获取失败 → 说明已经有另一个实例在运行，当前进程直接退出或等待

在分布式或多进程环境下，文件锁能防止多个进程同时修改状态、同时处理同一个内容，导致重复处理或数据错乱。它是一种轻量级的单实例保障。

防重复流程

一个典型的完整防重复流程如下：

  1. 获取单实例文件锁
    确保同一时间只有一个处理实例在运行。如果拿不到锁，直接退出。

  2. 检查尺寸 + 哈希
    计算文件的尺寸和哈希值，在去重记录表（比如 Redis、数据库、本地缓存）中查找是否存在该组合。

    • 若存在 → 判定为重复，结束处理
    • 若不存在 → 记录该尺寸+哈希，然后继续处理

  3. 5秒冷却检查
    对于刚刚处理过的内容，在去重记录表里额外记录一个”处理时间戳”。如果当前时间与上次处理时间差 < 5 秒，则直接拒绝（即使尺寸+哈希相同）。
    实际中常把冷却时间整合到去重记录的过期时间中（例如键值对设置 TTL = 5 秒）。

  4. 执行业务处理（如下载、入库、通知等）

  5. 更新冷却标记
    设置该内容的最后处理时间为当前时间，并保留 5 秒。

AI agent的看法

我认为agent主要就是一个可以进行思考，规划，使用工具且执行行动后从得到的结果中获取反馈的一个ai
Agent = LLM大模型(类似大脑) + 规划的能力 + 能使用工具 + 行动的接口
规划能力：是可以将我们提出的复杂问题拆分成解决问题的每一个优先级步骤
工具使用：能调用搜索引擎，API，等等外部的工具
执行行动：在我们规划好的范围内，人为不进行干预，让ai自主完成实现代码的运行

其中agent有一些重要的理论比如：React架构，TAO循环
React架构：(reasoning + acting,推理加行动)，就是让ai不只是拥有文本输出，而是像人一样边思考边行动
TAO循环：拥有三个主要的模块Thought(思考)、Action(行动)、Observation(观测)
Thought(思考)：ai接受到人类指令时，会进行思考然后计划下一步开始做什么，并且分析当前的状况，推理出需要的信息，最后决定采取什么行动
Action(行动)：ai在思考中进行的决定，它会调用搜索工具，查询SQL，或者进行计算，与外部进行互动
Observation(观测) ：ai接收并分析行动的反馈结果，如果结果不理想，循环就会再次启动，进行新一轮的思考-行动-观测

React架构的本质是推理-行动-观测三者的循环迭代
React架构的设计理念就是定义了一个让AI能够思考-行动-观测的行为模式
TAO循环是实现ReAct的具体运转步骤，是这个模式的代称

Guardrails(护栏)：强大的ai需要设置安全的边界，防止ai失控
react-cycle-guard :它正是React循环与护栏概念在代码层面的结合,一个用来检测和防止React组件陷入无限渲染循环的护栏，而且能能确保UI的稳定
总结：TAO循环是让AI变“活”的运转模式，护栏是确保AI变“稳”的安全机制。
