Session

Session 是服务端存的数据，用来记住”这个用户是谁、登录了没有”。例如：

特点：

  • 存在服务器端（Flask 默认加密存在 Cookie 里，也可以存 Redis/数据库）
  • 用户拿不到、改不了，安全
  • 登出时 session.clear() 删掉即可

Cookie + Session 的配合流程：

  1. 用户输入密码登录
  2. 服务器验证通过 → 创建 Session（存用户信息）→ 返回一个 Session ID
  3. 浏览器把 Session ID 存到 Cookie 里
  4. 之后每次请求，浏览器自动在 Cookie 里带上 Session ID
  5. 服务器根据 Session ID 查到 Session 数据 → 知道是谁在操作

Redis

Redis 是一个内存数据库，数据存在内存里而不是硬盘上，所以读写极快。

Session 的存储方式之一就是 Redis：

存储方式 特点
Flask 默认（加密 Cookie） 简单，但数据量受限
Redis 快、支持分布式、服务器重启数据不丢
SQLite/MySQL 持久化，但比 Redis 慢

简单记住：Cookie 是浏览器存的通行证，Session 是服务器存的用户档案，Redis 是存放这些档案的高速柜子。

关键技术概念

哈希

哈希是一种将任意长度的数据（比如文件、文本、数字）转换成固定长度短字符串的算法。这个短字符串通常被称为哈希值，相当于给数据一个独一无二的”指纹”——相同的数据永远返回相同的哈希值，而且计算速度很快。

哈希的主要用途：

  • 数据去重：比较哈希值判断内容是否重复
  • 密码储存：只存哈希值，不存明文密码
  • 文件完整性校验：对比下载前后哈希值是否一致
  • 哈希表/字典：编程语言中的 Map、Object、Dict 都用哈希来快速定位键值对

单实例文件锁

单实例文件锁是一种并发控制/互斥机制，确保同一时间整个系统只有一个进程（或线程）在执行核心的去重+处理逻辑。它使用操作系统提供的文件锁（flock、LockFile 等）来锁住一个约定的锁文件。

进程启动时尝试获取锁：

  • 成功获取 → 成为单实例，继续执行
  • 获取失败 → 说明已经有另一个实例在运行，当前进程直接退出或等待

在分布式或多进程环境下，文件锁能防止多个进程同时修改状态、同时处理同一个内容，导致重复处理或数据错乱。它是一种轻量级的单实例保障。

防重复流程

一个典型的完整防重复流程如下：

  1. 获取单实例文件锁
    确保同一时间只有一个处理实例在运行。如果拿不到锁，直接退出。

  2. 检查尺寸 + 哈希
    计算文件的尺寸和哈希值，在去重记录表（比如 Redis、数据库、本地缓存）中查找是否存在该组合。

    • 若存在 → 判定为重复，结束处理
    • 若不存在 → 记录该尺寸+哈希，然后继续处理

  3. 5秒冷却检查
    对于刚刚处理过的内容，在去重记录表里额外记录一个”处理时间戳”。如果当前时间与上次处理时间差 < 5 秒，则直接拒绝（即使尺寸+哈希相同）。
    实际中常把冷却时间整合到去重记录的过期时间中（例如键值对设置 TTL = 5 秒）。

  4. 执行业务处理（如下载、入库、通知等）

  5. 更新冷却标记
    设置该内容的最后处理时间为当前时间，并保留 5 秒。

一个可以提取图片中重要信息并分成优先级的ai助手

项目概述

日常工作中截图里有很多待办信息，手动录入日历太麻烦，所以想用 AI 自动化。

项目的本质是：用户截图或上传图片 → OCR 提取文字 → 交给 AI 大模型分析总结 → 最终提取出待办事项。

技术选型

OCR（光学字符识别，Optical Character Recognition）是一种能把图片、扫描件或纸质文档上的文字”翻译”成电子设备可以编辑、搜索的文本的技术。

我们需要强大的 OCR 来处理图片，因此选择了 PaddleOCR。相比传统 OCR 工具，PaddleOCR 的优势在于：

  • 基于深度学习架构，识别精度很高
  • 支持的语言库庞大
  • 传统的 OCR 只能逐字读出文字，而 PaddleOCR-VL 系列模型可以看懂文档的复杂排版
  • 极其轻量化，部署成本低

下一步就是把提取出来的文字交给 AI 大模型分析，比如 DeepSeek、ChatGPT 等。需要注意的是，如果使用 ChatGPT 可以简化 OCR 步骤，因为它具有 Vision（视觉）模型，能直接从图片中识别文字。AI 分析后会返回 JSON 格式的结构化数据，例如：

返回 JSON 的好处是结构化——程序可以直接用 对象.属性 或 数组[索引] 的方式读取数据，前后端都能轻松处理，更加轻量化。

前端使用 Streamlit，它是一个开源的 Python 框架，可以快速将 Python 脚本转化为交互式 Web 应用。只需 pip install streamlit，然后使用内置函数就能创建页面和交互组件：

  • st.title：定义页面标题
  • st.button：触发交互操作
  • st.dataframe：展示核心数据表格

剪贴板与图片预处理

ImageGrab.grabclipboard() 是 Windows 专有 API，能直接从剪贴板拿到 PIL Image 对象——不需要保存文件到磁盘，也不需要用户选择路径。

MD5 去重：MD5 是一种哈希算法，能将任意长度的数据（比如一张图片、一段文字）计算成一个固定 128 位（32 个十六进制字符）的”指纹”。Windows 截图工具对同一张图不会重复放入剪贴板，但微信/QQ 截图可能会。对整张图的 tobytes() 做 MD5，比存 PIL Image 对象更省内存。

图片预处理（pil_to_base64）：

  • 长边超过 2048px 等比缩放，控制传输体积
  • RGBA/P 模式转 RGB，避免 JPEG 编码报错
  • quality=85，在体积和清晰度之间取平衡

processing 标志：防止 AI 分析期间（可能 5-10 秒）重复触发。外层 while True 捕获所有异常，内层 last_hash = None 重置确保出错后下一次能重新尝试。

Vision LLM 与 Prompt

关于 Vision LLM 的选择：刚开始试用的是 DeepSeek，后来发现它不是多模态模型，于是换成了阿里云的 qwen-vl-max。也可以使用 GPT-4V 等，但需要注意 GPT 的 Vision 模型需要网络代理，而且成本较高。

Prompt 的设计思路：

  • 逐区域扫描 → 列出每个任务 / @提及 / 会议 / 截止日期 / 代码 TODO
  • 即使只有模糊线索也要列出（对抗模型”偷懒”倾向）
  • 按 紧急 → 重要 → 普通 三级分类
  • 统一 Markdown 输出格式

后续在 Prompt 设计上更新了一些指令——“逐区域扫描”、”即使只有一条模糊的任务线索也要列出来”。这些指令是对模型”偷懒”倾向的对抗，实际测试中发现不加这些约束，模型经常直接回复”没有待办事项”。

日历同步

写入默认日历，无弹窗。

appt.Save() 直接入库，不像 .ics 方案需要弹出日历窗口让用户确认。事件会自动同步到手机——只要 iPhone/Android 上登录了同一个 Outlook/Exchange 账号。

COM 是微软的一套二进制接口标准，Windows 剪贴板、OLE 拖放、自动化等都建立在它之上。

设计原则

  • 零依赖云服务：不依赖任何外部账号体系。唯一的外部 API 是 DashScope（AI 分析）
  • 静默优先：后台运行时不弹控制台（clipboard_watcher.vbs 用 pythonw.exe 启动），成功/失败都走 Toast 通知，不打断用户工作流
  • 渐进降级：Outlook COM → .ics 文件 → 下载按钮。每条路径都有兜底，没有单点故障
  • 模块复用：calendar_sync.py 的 add_to_calendar() 同时服务于 hotkey_screenshot.py（后台监听）和 app.py（Web UI），一套逻辑两处复用

AI agent的看法

我认为agent主要就是一个可以进行思考，规划，使用工具且执行行动后从得到的结果中获取反馈的一个ai
Agent = LLM大模型(类似大脑) + 规划的能力 + 能使用工具 + 行动的接口
规划能力：是可以将我们提出的复杂问题拆分成解决问题的每一个优先级步骤
工具使用：能调用搜索引擎，API，等等外部的工具
执行行动：在我们规划好的范围内，人为不进行干预，让ai自主完成实现代码的运行

其中agent有一些重要的理论比如：React架构，TAO循环
React架构：(reasoning + acting,推理加行动)，就是让ai不只是拥有文本输出，而是像人一样边思考边行动
TAO循环：拥有三个主要的模块Thought(思考)、Action(行动)、Observation(观测)
Thought(思考)：ai接受到人类指令时，会进行思考然后计划下一步开始做什么，并且分析当前的状况，推理出需要的信息，最后决定采取什么行动
Action(行动)：ai在思考中进行的决定，它会调用搜索工具，查询SQL，或者进行计算，与外部进行互动
Observation(观测) ：ai接收并分析行动的反馈结果，如果结果不理想，循环就会再次启动，进行新一轮的思考-行动-观测

React架构的本质是推理-行动-观测三者的循环迭代
React架构的设计理念就是定义了一个让AI能够思考-行动-观测的行为模式
TAO循环是实现ReAct的具体运转步骤，是这个模式的代称

Guardrails(护栏)：强大的ai需要设置安全的边界，防止ai失控
react-cycle-guard :它正是React循环与护栏概念在代码层面的结合,一个用来检测和防止React组件陷入无限渲染循环的护栏，而且能能确保UI的稳定
总结：TAO循环是让AI变“活”的运转模式，护栏是确保AI变“稳”的安全机制。

git的基础语法

如何使用git
git的基本操作
pwd–打印当前的工作目录
cd–移动到目录
ls–列出目录下的文件
git –version–查看git的版本
git init–初始化一个git仓库
git status–查看当前仓库的状态
git add. 只需一个命令，将两个修改过的文件添加到备用区域。
git commit -m提交一个提交，包括合适的标志，这样你就可以作为命令的一部分提供日志消息
git log–查看提交历史
git log -2 仅查看最近两个提交的信息。
git log –since’–查看自某个日期以来的提交
git diff–查看文件的修改内容
git diff –staged–运行命令将最后提交的版本与预备区的版本进行比较
git diff –report.md–查看report.md文件的修改内容
git diff HEAD1 HEAD2–比较最近两次提交之间的差异
git revert–撤销一个提交
git checkout–切换分支或恢复工作树文件
git restore –staged–将文件从暂存区恢复到工作区
git revert HEAD –no-edit– 撤销最近的提交，并使用默认的提交消息
